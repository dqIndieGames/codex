## Checklist

1. `[修复] F4 snapshot 护栏修复`
   会话头与升级弹窗等直接展示 `local1` 版本链的 snapshot/断言必须改成与当前运行时 `-local1` 口径一致；不得只补说明文本而继续保留旧 snapshot 基线。

2. `[修复] F8 的 401 budget 语义回收`
   `/responses` 主链上的 `401` 不再允许先进入 unauthorized recovery follow-up 语义再回到普通 retry；必须直接落入既有普通 retry budget。此次修复只改变 `401` 的控制流与 budget 归属，不改变 bounded/unbounded 默认语义，不移除端点级 `max_attempts(1)` 例外，不改变预算耗尽后的原有终态分流。

3. `[修复] F12 的 401 控制流统一`
   request-layer、turn-level stream/websocket、websocket reconnect、fallback to HTTP 等相关路径都必须统一直走普通 retry；不得残留任何单独的 `401` unauthorized recovery 优先分支，也不得把 `401` 改成直接终止。

4. `[风险收敛] F9 的分类入口统一`
   route/path 真值必须由请求入口通过 `endpoint/session` 显式透传到 telemetry/suppress 主链；禁止新增基于 URL 猜测的分支、局部 websocket 白名单、散落的 ad-hoc `if should_retry`、或通过全局宏/一刀切静音来替代结构收敛。

5. `[保护] F1-F3 local1 显示链不回归`
   CLI/TUI 版本号、状态卡片、顶部状态区、历史单元、升级提示、版本跳转提示继续统一保持 `-local1` 版本身份；后续修复不得顺带把显示链回退成官方裸版本。

6. `[保护] F5-F7 retry 主链行为不回归`
   非 `/responses` whitelist、`402 usage-limit` 区分、history cleanliness、metrics 保留、retry 详情保留、单次退避上限 `10s` 等既有通过项都必须保持；修复 `401` 时不得顺带放宽或收窄其他 HTTP 分类。

7. `[保护] F10 的 RPC 层与零线程成功口径`
   `thread/providerRuntime/refresh` 与 `thread/providerRuntime/refreshAllLoaded` 两个入口必须保留；`refreshAllLoaded` 在 `totalThreads = 0` 时仍返回非失败，`failedThreads = []` 继续视为成功；空闲线程仍保留 `Applied/Queued` 或等价语义，不得误报失败。

8. `[保护] F10 的运行时 refresh 消费边界`
   active thread 收到 refresh 后，后续 `turn-level retry`、`websocket reconnect`、`fallback to HTTP`、`request-layer internal retry` 必须在下一次 retry 前吃到最新 refresh；若当前 attempt 尚未进入下一次 retry，仍不承诺中途热切，不得借本轮修复擅自改变这条边界。

9. `[保护] F10 的配置解析与 Windows tray/control plane 语义`
   `[agents.*].config_file` 的相对路径继续按当前 user `config.toml` 所在目录解析；Windows tray/control plane 的 provider copy/replace 仍只复制 `base_url` 与 `experimental_bearer_token`，不得误切换 `model_provider_id`；目标 provider 不存在、source 缺字段、catalog 不一致、配置写入失败等场景必须明确失败，不允许静默落盘。

10. `[保护] F11 的历史发现与继续会话语义`
    `thread/list` 在未传、`null`、空数组这三种 `modelProviders` 输入下都等价于不过滤；显式 provider 过滤继续生效；返回项继续保留 `modelProvider` 字段。CLI `resume --last` 与本地 TUI resume picker 默认不按 provider 收窄；remote TUI 继续维持“不从客户端注入 provider”的现状。

11. `[保护] F13 与 A2 的 gpt-5.4 priority fallback 口径`
    顶层 `force_gpt54_priority_fallback` 省略或显式 `true` 时，`gpt-5.4` 仍强制 `priority`；显式 `false` 时，不只关闭 `priority` 兜底，还要同时关闭 `Fast` 透传；`Flex` 继续保留；非 `gpt-5.4` 不受影响；任何 `[profiles.*].force_gpt54_priority_fallback` 都不得覆盖顶层。

12. `[保护] F14-F15 默认日志过滤与日志能力不回归`
    Windows app/app-server 与 TUI 默认日志过滤继续保持 `warn`；显式 `RUST_LOG` 覆盖继续生效；`codex-tui.log`、sqlite log 能力与现有线程生命周期、协议语义不得被本轮修复改坏。

13. `[保护] A1 首次对话固定清单 once-only 语义`
    fixed local1 清单继续只在新线程第一条用户消息后插入一次；resume、continue、历史线程重开均不得重复插入；refresh/retry + Windows tray 联动概述行继续存在，且文案语义不得漂移。

14. `[边界] 后续实现不得扩大到无关功能`
    不得借本轮修复顺带调整无关 provider 行为、无关日志策略、无关 UI 文案、无关历史筛选逻辑、无关 API、无关配置字段或其他未在本文冻结范围内的功能。

## Acceptance

1. `401` 控制流验收
   `/responses` 主链中的任意 `401` 场景都不再优先走 unauthorized recovery，而是直接进入普通 retry；request-layer、stream/websocket、websocket reconnect、fallback to HTTP 全部一致；bounded/unbounded 默认语义、端点级 `max_attempts(1)` 例外、预算耗尽终态与错误分类扩展继续分离。

2. snapshot/断言基线验收
   与 `local1` 版本展示直接相关的 snapshot、断言与测试护栏全部与当前运行时 `-local1` 口径一致；`F4` 被修复后，不再存在会话头或升级弹窗仍显示旧值的漂移。

3. suppress/telemetry 验收
   retry/reconnect suppress 统一由单一主链分类入口驱动；route/path 真值来自显式透传，不依赖 URL 猜测；`codex.api_request`、`codex.websocket_connect`、`codex.websocket_request` 与失败型 `codex.websocket_event` 的中间态 suppress 继续有效；metrics、`Reconnecting... N`、`additional_details` 与 UI retry 详情继续保留。

4. retry 保护性行为验收
   非 `/responses` 的旧 whitelist、`402 usage-limit` 区分、`10s` 单次 backoff 上限、history cleanliness、最终失败终态路径都不被回归；修复 `401` 后，其他已通过状态分类继续保持原口径。

5. provider runtime / Windows tray 验收
   `refresh`/`refreshAllLoaded` 两个 RPC 继续存在；`totalThreads = 0` 时仍非失败，`failedThreads = []` 继续视为成功；空闲线程 `Applied/Queued` 语义保留；active thread 后续 retry 边界继续能吃到最新 refresh；`[agents.*].config_file` 相对路径继续按 user `config.toml` 目录解析；Windows tray/control plane 继续只复制两字段、不切换 `model_provider_id`，并对字段缺失、目标不存在、catalog 不一致、写入失败等场景给出明确失败反馈。

6. history / resume 验收
   `thread/list` 在未传、`null`、`[]` 的 `modelProviders` 输入下都默认返回全 provider；显式 provider 时仍过滤；返回项继续保留 `modelProvider`。CLI `resume --last` 与本地 TUI resume picker 默认不按 provider 收窄；remote TUI 继续保持现状，不自动切换到历史线程记录的 provider。

7. priority fallback / A2 验收
   顶层开关省略或显式 `true` 时，`gpt-5.4` 继续强制 `priority`；显式 `false` 且 `Fast` 时，不透传 `service_tier`；显式 `false` 且 `Flex` 时，`Flex` 继续保留；非 `gpt-5.4` 不受影响；任何 `[profiles.*].force_gpt54_priority_fallback` 都不得生效。

8. logging / A1 验收
   Windows app/app-server 与 TUI 默认日志过滤继续为 `warn`，显式 `RUST_LOG` 覆盖能力仍在；A1 继续只在新线程第一条用户消息后插入一次固定清单，resume/continue/历史线程重开不重复，refresh/retry + Windows tray 概述行仍存在且语义不漂移。

9. 文档完整性验收
   仅凭本文即可理解本轮问题、范围、修复目标、边界、验收方式与不回归要求；`F1-F15`、`A1-A2` 全部出现；`F4/F8/F12` 被明确标记为修复项，`F9` 被明确标记为风险收敛项，其余通过项均被明确写为保护性要求。
